# Telegram messages redesign summary

## Цель

Проведён аудит пользовательских Telegram-сообщений, которые выглядели как технический лог:
длинные строки с датой, маркетплейсом, моделью, внешним ID и статусом без визуальной иерархии.

Главная цель этапа — привести сообщения к единому аккуратному формату:

- заголовок с тематическим эмодзи;
- короткое пояснение;
- карточные блоки вместо дампа строк;
- единые статусы;
- безопасный HTML;
- читаемые даты, суммы и проценты.

## Что было переработано

### Последние заказы

Старый формат:

```text
— 16.05.2026 18:19 OZON FBS #16039101-0748-4: требует обработки
```

Новый формат:

```text
🛒 <b>Последние заказы</b>

Показываю 10 последних заказов по всем кабинетам.

1. ⚠️ <b>Ozon · FBS</b>
   • Дата: 16.05.2026, 21:19
   • Заказ: <code>16039101-0748-4</code>
   • Статус: Требует обработки
```

Длинные ID сокращаются только визуально в списках. Полные значения остаются доступными в
детальной карточке заказа.

### Сводка

`DailyReportService` теперь формирует более читаемую сводку:

- блок `Продажи и заказы`;
- блок `Финансы`;
- блок `По маркетплейсам`;
- списки через `•` вместо смешения `—` и простого текста.

### Остатки и out-of-stock

Сообщения остатков переведены на карточки:

- товар / артикул;
- остаток;
- склад;
- прогноз;
- потенциальные потери.

### FBS / rFBS контроль

Предупреждения о риске просрочки теперь показывают:

- человекочитаемый маркетплейс;
- модель продаж;
- короткий ID заказа;
- дедлайн отдельной строкой.

### Ошибки синхронизации

Список ошибок синхронизации теперь оформляется как набор блоков:

- кабинет;
- ошибка;
- тип проблемы;
- понятная рекомендация.

### Empty states

Добавлен единый formatter для пустых состояний:

- `Заказов пока нет`;
- `Остатки пока не загружены`;
- `Прибыль пока не рассчитана`;
- `Ошибок синхронизации нет`.

## Новые formatter-модули

Добавлены:

- `app/bot/formatters/__init__.py`;
- `app/bot/formatters/common.py`.

Formatter не ходит в БД и не выполняет бизнес-логику. Handler получает данные, затем вызывает
готовый builder текста.

## HTML и безопасность

Добавлены и покрыты тестами:

- экранирование внешних значений через HTML escaping;
- безопасное отображение значений вроде `Крем <новый> & тест`;
- отсутствие старого технического формата `— дата MP FBO #id: статус`;
- форматирование ID заказа через `<code>`.

Глобальный `parse_mode=HTML` уже используется ботом; новые сообщения рассчитаны на HTML-разметку.

## WEB-проверка

Для WEB дополнительно прогнан regression smoke-test:

- `/web/login?token=...`;
- установка session cookie;
- redirect на `/web/`;
- рендер пустого FREE-dashboard без `Internal Server Error`.

Локально этот сценарий проходит. Если live-ссылка всё ещё отдаёт 500 после релиза `1.6.3`, первым
делом нужно проверить применение миграции `20260517_0013_subscription_lifecycle.py`, потому что
она добавляет поле `user_subscriptions.period`, используемое текущей моделью.

## Тесты

Добавлены проверки:

- formatter последних заказов использует карточную структуру;
- длинные ID сокращаются;
- внешние значения экранируются;
- empty state имеет единый вид;
- остатки отображаются блоками;
- FBS deadline alert не похож на технический лог;
- ежедневная сводка содержит новые секции.
